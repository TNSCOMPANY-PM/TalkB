## 🏗️ 시스템 아키텍처 (확정 방향)

### 핵심 원칙: 데이터 중심 분리

```
[트래킹 시스템 (TJ)]
    │ 외부 API → 측정 결과 → DB 저장
    ↓
[데이터베이스 (Supabase)]
    │ 단일 진실의 원천
    │ 양쪽 시스템이 데이터로만 통신
    ↓
[분석·결정 로직 (TJ)]
    │ DB 읽기 → "이 매장에 어떤 액션 필요?" 결정
    │ 작업 큐(Queue)에 작업 등록
    ↓
[작업 큐 (Supabase tasks 테이블)]
    │ 시스템 간 분리의 핵심
    │ 비동기 처리, 재시도 가능
    ↓
[실행 시스템 (박재민)]
    │ 작업 큐 폴링 → 작업 처리
    │ 콘텐츠 자동 생성·발행
    │ 실행 결과 DB 기록
```

### 절대 금기
- ❌ TJ 시스템이 박재민 시스템을 직접 호출
- ❌ 같은 정보를 두 시스템이 각자 저장
- ❌ 동기 통신 (즉시 응답 기다리기)

### 핵심 데이터 스키마 (시니어 자문 검증 필요)

```
stores (매장 정보)
queries (측정 쿼리, 매장별 50/200개)
measurements (측정 결과, 시간별)
tasks (작업 큐)
executions (실행 결과)
users (사용자, 가입자)
subscriptions (구독 정보)
```

**⚠ 데이터 스키마는 시니어 자문 영입 후 첫 작업으로 진행**
